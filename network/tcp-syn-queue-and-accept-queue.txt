The TCP Three-Way Handshake

Before the queues make sense, remember the handshake:

SYN: Client sends a connection request.

SYN-ACK: Server acknowledges and waits.

ACK: Client confirms, and the connection is "Established."




1. The SYN Queue (Incomplete Connection Queue)
This is the first waiting room. When the server receives a SYN packet, it creates an entry in this queue and sends back a SYN-ACK.

Status: The connection is "Half-Open."

Purpose: It tracks connections that haven't finished the handshake yet.

Limit: Controlled by the kernel parameter net.ipv4.tcp_max_syn_backlog.

What happens if it's full? The server may start dropping new SYN packets or trigger SYN Cookies (a mechanism to prevent SYN flood attacks by not storing state in the queue).




2. The Accept Queue (Completed Connection Queue)
Once the client sends the final ACK, the connection is fully established. The kernel moves the connection from the SYN queue to the Accept Queue.

Status: The connection is "Established."

Purpose: It holds connections that are ready and waiting for your application (like Nginx, Python, or Go) to call the accept() system call.

Limit: Controlled by the backlog argument in the application's listen() call and the system-wide net.core.somaxconn.

What happens if it's full? The server usually ignores the final ACK from the client. The client will think the connection is open, but the server won't respond to data, eventually leading to a "Connection timed out" or a "Connection refused" if configured that way.





How to check your queues

If you suspect your server is dropping connections because these queues are full, you can use the ss (socket statistics) command:

Recv-Q: The number of connections currently in the Accept Queue waiting to be accepted by the app.

Send-Q: The maximum size of the Accept Queue.

Pro Tip: If Recv-Q is consistently equal to Send-Q, your application is likely "blocking" or too slow to process incoming users, and you're dropping connections!
